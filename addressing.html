Addressing
Time & Accuracy

Overview

For Honestbee on demand service, addressing feature is the key basis of getting service delivered.
In this project, I recognised users' problems through observations, collected data that helps validate the iterations, and proposed to implement a new collection of Addressing experience that serves better across different countries in Southeast Asia.

Impact

Through the improvement, the addressing experience was more efficient. Information input of user was more accurate.
Address failure rate in decreased from 10% to avg.4%, which equals to 1960 orders, SGD$200,000 revenue of the organization per month across countries. It’s a huge impact in business value.

Honestbee Website

Problem

In Singapore, 10.67% of users are dropping off when they enter their address to start shopping. Users can't find their address served even when we do serve their address.
On Honestbee, the product showing to users will based on user’s address input. E.g., As a user, I want to see if my area is available for shopping, I put in Santosa (a served area), the system replies that the area is not served. So I drop off the website.

Goal

Efficiency & accuracy - to allow user a fluent shopping experience, from shopping to receiving, without hassling around by entering address

Research

As we were expanding our service to other cities, recognising the divergence in address from different culture was important. We did two research :

  1. Address structure analysis to understand divergence in different countries
  2. User studies to understand how user from different countries reacts to address questions.

Insight

I. Postal code that everyone uses v.s postal code that no body knows

Countries like Singapore, everyone knows and uses postal code, whereas countries like Philippines and Malaysia, no body knows postal code

II. Building name = address

Building name as an address is common in Thailand & Malaysia

III. Technical barriers

In most south-east asian countries, like Indonesia, Thailand, Malaysia & Philippines, Google map doesn't perform well in full address (it only works well to a certain level of address structure hierarchy)

Information Plan

When user type in serviceable area, we populate data to allow the automation of navigating the route for our deliverer. Later user complete the address with additional information manually in checkout, we use the information to allow deliverers to find the exact location.

Design iteration

Design Highlights

I. Content that matches the cultural difference

We came out with a set of content hints to guide user type in for different countries, to guide user type in accuraterly to populate the closest location

II. Progressive reveal UI

For countries that we called in multiple cities to prevent duplicated road names showing up from autocomplete. Because those countries, the same road name could exist in different cities. By using this approach, there's no confusion for users to choose from the list.

III. Double confirmation when needed

When user input an area, such as district or town, we double confirm, asking for dropping a pin on map, to ensure they provide a location to our system

Key UI Design

Efficiency & accuracy - to allow user a fluent shopping experience, from shopping to receiving, without hassling around by entering address

Prototype

Impact

We conducted a survey that mimics 80% of the user testing, and gained 1200 responses to reach 90% confidence of data outcomes.

I. How user would interact with the new address design

Avg 60.8% user would start entering their address; avg 25.5% user would click “Use my location”; and ,11.5% would interact with the map. Different elements serve for different senarios. This helped us to know why all the elements are designed.

II. How intuitive is progressive reveal UI

More than 50% of users mentioned they would click "City" column by just looking at the static UI. This shows that changing city interaction is expected to be easily understandable and learnable.

III. How successful the new design help reducing the “address isn’t served” error rate

"Address isn't served" error rate decreased to avg.4%, equivalent to 1960 orders per month across countries in South East Asia.

We are very proud to see the success. Now it’s fluently running on Honestbee websites across countries in South East Asia.

Final thoughts

Designers are problem solvers, we need to understand whys and hows in people to create values through qualifying the user experience, such as user research, observation, user testing. But we also need to scientifically understand what actually works and what doesn’t through quantifying the user experience, such as survey and A/B testing. All of which also clears ourself out, to face the challenging questions of why one is better than another, further how it impacts money value in business.
